## Project Goal

Create an **educational terminal-based RPG** that teaches players about:
- Indonesian history (10 kingdoms, 7th-17th century)
- Indonesian mythology (22 creatures: Genderuwo, Kuntilanak, Tuyul, etc.)

**Target Audience:** International players (ages 13+)
**Language:** English with Indonesian cultural terms explained
**Platform:** Terminal/CLI (Windows, macOS, Linux)

## Critical Issues Found & "Fixed" (NOT TESTED)

### Issue #1: Enemy Inconsistency (CRITICAL)

**Problem:** Three different enemies appear in one encounter:

1. **AI Story Description:** "A massive Naga emerges from the depths..."
2. **Enemy Display Name:** "ENEMY: Kuntilanak"
3. **Actual Combat Enemy:** Random enemy from `Enemy.create_random()` (could be Tuyul)

**Root Cause:**

```python
# trpg/game.py - start_combat()
enemy = Enemy.create_random(self.player.level)  # Creates random enemy (e.g., Tuyul)

encounter_text, enemy_hint = self.ai_engine.generate_encounter(...)  # AI says "Naga"

# enemy_hint is IGNORED! Combat uses the random enemy, not the AI's creature
```

**Current "Fix" Applied:** None - still broken

**Why It's Broken:**
- AI generates encounter text with creature name
- Game creates random enemy from `ENEMY_TEMPLATES` list
- `enemy_hint` from AI is captured but never used
- Player sees: Story says "Naga", UI shows "Kuntilanak", combat is against "Tuyul"

**Request for Better Solution:**

We need ONE of these approaches:

**Option A: AI Generates Enemy Stats**
```python
# AI returns enemy data along with description
{
    "description": "A massive Naga emerges...",
    "enemy_name": "Naga",
    "enemy_hp": 45,
    "enemy_attack": 14,
    "enemy_defense": 6
}
# Game creates enemy from AI data
```

**Option B: Match AI Hint to Enemy Template**
```python
# AI returns hint: "naga"
# Game searches ENEMY_TEMPLATES for matching name
# If found, use that template; if not, use random
enemy_hint = "naga"
if enemy_hint in [e["name"].lower() for e in ENEMY_TEMPLATES]:
    enemy = Enemy.create_from_template(enemy_hint)
else:
    enemy = Enemy.create_random(level)
```

**Option C: Pass Enemy to AI**
```python
# Game creates enemy FIRST
enemy = Enemy.create_random(level)

# Then tell AI which enemy to describe
encounter_text = ai_engine.generate_encounter(
    player_context,
    location,
    forced_enemy_name=enemy.name  # AI must describe THIS enemy
)
```

**Which approach is best?** We need advice on:
- Maintaining game balance (AI shouldn't create overpowered enemies)
- Keeping AI creative (descriptions should be varied)
- Performance (minimal API calls)

### Issue #2: Equipment System Not Working (CRITICAL)

**Problem:** Players cannot equip items. Items sit in inventory but can't be equipped.

**Current Flow:**
```python
# Player opens equipment screen
# Presses 'E' to equip
# Types "keris kecil"
# Game calls: self.player.equip_item("keris kecil")
# Returns: (False, "You can't equip materials.")
```

**Root Cause:**

Item category mismatch. Starting items might be created with wrong category:

```python
# trpg/player.py - Item.__init__()
if self.name in Item.ITEM_DATABASE:
    data = Item.ITEM_DATABASE[self.name]
    self.category = data["category"]  # Should be WEAPON
else:
    self.category = ItemCategory.MATERIAL  # Default - CAN'T EQUIP!
```

**Current "Fix" Applied:** Verified all starting_items use correct ITEM_DATABASE names.

**Why It Might Still Be Broken:**
- Item name matching is case-sensitive (`.lower()` should fix)
- Item might not exist in ITEM_DATABASE
- Equipment UI flow might have bugs

**Request for Better Solution:**

We need:
1. **Debug logging** to see why equip_item() returns False
2. **Better error messages** ("Can't equip" → "This is a consumable, not a weapon")
3. **UI improvements** - Show item category in inventory
4. **Auto-equip suggestions** - "You found Keris Kecil! Equip it? [Y/n]"

**Specific Questions:**
- Should equipment be a separate screen or inline?
- Should items auto-equip if slot is empty?
- Should there be an "equip best" feature?

### Issue #3: Story Teleportation (HIGH)

**Problem:** AI teleports player between locations instantly.

**Example:**
```
Turn 1: "You examine the heirloom storage" (indoors)
Player choice: "Inspect the treasury"
Turn 2: "As you wade through the murky river..." (outdoors - teleport!)
```

**Current "Fix" Applied:**

Added to AI prompt:
```
CRITICAL: CONTINUE from Scene X. DO NOT restart the story.
DO NOT teleport to a new location. Stay in the current scene
unless the player explicitly travels.
```

**Why It Might Not Work:**
- AI doesn't understand spatial relationships
- AI prioritizes "exciting" over "consistent"
- No penalty for breaking continuity

**Request for Better Solution:**

**Option A: Location State Tracking**
```python
# Track player location explicitly
self.current_location = {
    "name": "Heirloom Storage",
    "type": "indoor",
    "parent": "Palace",
    "exits": ["corridor", "courtyard"]
}

# Include in every AI prompt
CURRENT LOCATION: Heirloom Storage (indoor, part of Palace)
AVAILABLE EXITS: corridor, courtyard

# Validate AI output
if "river" in ai_response and current_location["type"] == "indoor":
    reject_and_regenerate()
```

**Option B: Transition Scenes**
```python
# Force transition scenes when location changes
if player_choice == "Go to river":
    # First scene: walking to river
    scene = "You leave the storage, walk through corridors, exit the palace..."
    # Next turn: at river
    location = "River"
```

**Option C: Post-Processing Validation**
```python
# After AI generates scene, validate consistency
def validate_scene_consistency(scene, previous_location):
    if "river" in scene and previous_location == "indoor":
        return False, "Teleportation detected"
    return True, None
```

**Which approach is best?** We need advice on:
- Minimizing API calls (validation = extra calls?)
- Keeping AI creative (too many constraints?)
- Implementation complexity

### Issue #5: AI Output Format Leaking (MEDIUM)

**Problem:** AI includes "NARRATIVE SECTION:" and "CHOICES SECTION:" in output.

**Current "Fix" Applied:**

Updated SYSTEM_PROMPT:
```
FORMAT RULES:
- Output ONLY the story narrative (80-150 words)
- Output EXACTLY 3 numbered choices at the end
- DO NOT include section headers like "NARRATIVE SECTION" or "CHOICES SECTION"
```

**Why It Might Not Work:**
- AI sometimes ignores format instructions
- Temperature too high (0.85) causes creativity over rules
- No validation/parsing of AI output

**Request for Better Solution:**

**Option A: Output Parsing**
```python
# Strip known bad patterns
response = response.replace("NARRATIVE SECTION:", "")
response = response.replace("CHOICES SECTION:", "")

# Or use regex to extract only story text
story_match = re.search(r'(Year \d+ CE\..*?)(?=Choices:|$)', response, re.DOTALL)
```

**Option B: Structured Output**
```python
# Ask AI for JSON
{
    "narrative": "Year 1350 CE. You stand...",
    "choices": ["Choice 1", "Choice 2", "Choice 3"]
}
# Parse JSON, ignore extra text
```

**Option C: Lower Temperature**
```python
# Reduce creativity, increase rule-following
temperature=0.7  # Instead of 0.85
```

**Which is best?** JSON parsing might be most reliable but costs more tokens.

### Issue #6: Choice Length & Quality (MEDIUM)

**Problem:** AI generates long, multi-action choices.

**Example (Bad):**
```
"Disguise yourself as a merchant and infiltrate the alliance"
```
- 9 words
- 3 actions (disguise + travel + infiltrate)
- Assumes knowledge player doesn't have

**Current "Fix" Applied:**

Starting points rewritten with short choices (3-6 words). AI prompt includes:
```
CHOICE RULES:
- Each choice must be ONE sentence ONLY (10-25 words)
- NO markdown formatting (no **, *, _, etc.)
```

**Why It Might Not Work:**
- AI ignores word count instructions
- No validation of choice length
- Starting points fixed, but AI-generated choices not

**Request for Better Solution:**

**Option A: Post-Processing Validation**
```python
def validate_choices(choices):
    for choice in choices:
        if len(choice.split()) > 8:
            return False, "Choice too long"
        if " and " in choice:
            return False, "Multiple actions"
    return True, None
```

**Option B: Few-Shot Examples**
```python
# Include examples in prompt
GOOD EXAMPLES:
- "Question the merchants"
- "Inspect the treasury"
- "Follow the guard"

BAD EXAMPLES:
- "Disguise yourself as a merchant and travel to the inland kingdom to infiltrate the secret alliance"
```

**Option C: Choice Rewriting**
```python
# If AI returns bad choices, rewrite them
if len(choice.split()) > 8:
    choice = rewrite_choice_shorter(choice)
```

## System Architecture Issues

### Problem: Monolithic Game Class

**File:** `trpg/game.py` (1046 lines)

**Issue:** Single class handles everything:
- Game loop
- Combat
- Shop
- Save/Load
- UI
- Story management

**Request:** Should we refactor to ECS (Entity-Component-System) or keep it simple?

**Trade-offs:**
- **Current:** Easy to understand, hard to extend
- **ECS:** Harder to understand, easier to add features

---

### Problem: No State Machine

**Issue:** Game state is tracked via boolean flags:
```python
self.in_combat = False
self.in_shop = False
self.game_over = False
self.game_ended = False
```

**Problem:** Invalid states possible (in_combat AND in_shop = True)

**Request:** Should we implement proper state machine?

```python
class GameState(Enum):
    EXPLORATION = "exploration"
    COMBAT = "combat"
    SHOP = "shop"
    GAME_OVER = "game_over"

self.state = GameState.EXPLORATION
```

---

### Problem: AI Context Management

**Issue:** 50-scene history sent every turn.

**Current approach:**
```python
recent_history = self.story_history[-50:]
for i, scene in enumerate(recent_history, 1):
    prompt_parts.append(f"\n[Scene {i}]: {scene}")
```

**Problems:**
- Token usage grows with history
- AI might lose important details in 50 scenes
- No prioritization (combat scene = same weight as rest scene)

**Request:** Better context management strategy?

**Options:**
1. **Summary + Recent:** Summarize old scenes, keep last 5 full
2. **Key Events Only:** Track only major plot points
3. **Embedding Search:** Search for relevant past scenes
4. **Keep Current:** 50 scenes works fine
